# NetworkRadar Pro：IP 纯净度与底部评级板块改造方案

> 目标文件：`Egern/Widget/NetworkRadar-Pro.js`  
> 当前本地版本：`1.3.2`  
> 计划实现版本：`1.3.3`  
> 方案日期：2026-07-19

---

## 1. 方案摘要

本次改造将重新定义 NetworkRadar Pro 最后一个评级板块：

- **主分数改为 IP 纯净分**，范围为 `0～100`。
- **IPPure 的 `fraudScore` 作为主数据来源**。
- 纯净分基础公式为：

```text
纯净分 = 100 - IPPure fraudScore
```

- `ipapi.is` 的 `abuser_score` 和 `proxycheck.io` 的 Proxy、VPN、Tor、Abuser、Risk 信息只用于风险修正与兜底，不与 IPPure 简单平均。
- ISP、ASN、住宅/移动/机房属性作为解释信息，不直接替代 IP 纯净度。
- 延迟、IPv6、QUIC、NAT、DNS、流媒体和 AI 解锁不再参与纯净等级，避免把“线路不好用”错误解释为“IP 不干净”。
- 底部 UI 重构为三层：
  1. 纯净等级、纯净分和风险标签；
  2. IPPure、IP 属性、Abuser 三个核心数据块；
  3. ISP、地区、延迟、服务通过数量等辅助摘要。

最终目标是让用户一眼分清：

```text
IP 是否干净
IP 属于什么类型
风险判断是否可信
节点线路和服务是否好用
```

## 3. 当前脚本存在的问题

目标文件：

```text
D:\cc code\egrn\rule-for-mihoyo\Egern\Widget\NetworkRadar-Pro.js
```

### 3.1 存在两套互相冲突的评级逻辑

当前同时存在：

```js
nodeQualityRating()
rankNode()
```

`nodeQualityRating()` 使用 ISP、IP 属性、信誉、网络、服务等维度计算一次总分；随后 `rankNode()` 又从 100 分开始重新扣分。

底部 UI 最终展示的是 `rankNode()`，导致第一套评分中的 ISP 子分、风险子分和网络子分没有完整反映到最终界面。

改造后：

- 删除或停用两套重复总分逻辑。
- 使用唯一的 `calculatePurityRating()` 生成底部评级数据。
- 网络和服务保留独立摘要，不再参与纯净等级。

### 3.2 `abuser_score` 正则表达式错误

当前类似：

```js
.match(/([0-9.]+)s*(([^)]+))/)
```

应修正为：

```js
.match(/([0-9.]+)\s*\(([^)]+)\)/)
```

并处理不同数值范围：

```js
function normalizePercent(value) {
  const n = Number(value);
  if (!Number.isFinite(n)) return null;
  return n <= 1 ? n * 100 : n;
}
```

例如：

```text
0.45 (Elevated) → 45%
45 (Elevated)   → 45%
```

### 3.3 Proxy/VPN 被错误推断为住宅 IP

当前分类中存在类似逻辑：

```js
if (flags.proxy || flags.vpn) {
  return "住宅 IP";
}
```

Proxy/VPN 只能说明风险或用途，不能证明是住宅网络。

改造后分离为：

```text
IP 属性：住宅 / 移动 / 机房 / 未知
风险标记：Proxy / VPN / Tor / Abuser
```

### 3.4 综合质量与风险等级混在一起

当前可能因为延迟高、IPv6 不可用或服务解锁少而显示“高风险”。

但这些只能说明节点不好用，不能说明 IP 有欺诈或滥用风险。

改造后：

- 纯净等级只由 IP 风险数据决定。
- 节点延迟只显示在线路摘要中。
- 解锁数量只显示在服务摘要中。
- 不再用综合分推导“低风险 / 中风险 / 高风险”。

### 3.5 API 失败可能被当作安全

风险 API 获取失败时，不能使用 `0` 作为风险值，因为：

```text
未知 ≠ 低风险
```

改造后统一使用三态数据：

```text
成功：有真实数值
失败：待核验
缺失：不参与评分
```

## 4. 改造目标与非目标

### 4.1 改造目标

1. 给出真实、可解释的 IP 纯净分。
2. IPPure 为主来源，其他来源进行修正和兜底。
3. 风险 API 失败时明确显示“待核验”。
4. 纯净等级与线路质量、解锁能力分开。
5. 底部 UI 能展示分数来源及判断依据。
6. 不展示没有真实请求结果的数据源。
7. 保持 Egern Medium Widget 下信息清晰，不增加明显的纵向空白。

### 4.2 非目标

本次不做以下事情：

- 不把 ISP 品牌直接等同于 IP 纯净度。
- 不建立复杂且难以维护的全球 ASN 排名。
- 不因为 AWS、Oracle、Vultr 等机房 ISP 就直接判定高风险。
- 不因为住宅 IP 就直接判定满分。
- 不让流媒体或 AI 解锁结果影响纯净等级。
- 不增加没有真实数据支撑的风险来源。

## 7. 核心评分算法

## 7.1 基础纯净分

IPPure 成功时：

```js
baseRisk = clamp(ippure.fraudScore, 0, 100);
basePurity = 100 - baseRisk;
```

例如：

```text
fraudScore = 8
basePurity = 92
```

### 7.2 ipapi.is Abuser 修正

建议规则：

| Abuser 情况 | 修正 |
|---|---:|
| Very Low / Low，且小于 20% | 0 |
| Elevated / Moderate，或 20%～39% | -5 |
| High，或 40%～69% | -15 |
| Very High，或不低于 70% | -25 |
| `is_abuser === true` | 最终分不得高于 35 |

示例：

```js
if (abuserPercent >= 70 || level === "very high") score -= 25;
else if (abuserPercent >= 40 || level === "high") score -= 15;
else if (abuserPercent >= 20 || isElevated(level)) score -= 5;
```

### 7.3 proxycheck.io 修正

建议规则：

| 检测结果 | 处理 |
|---|---|
| Tor | 最终纯净分上限 10 |
| 明确 Abuser/Blacklist | 最终纯净分上限 25 |
| Proxy 与 VPN 同时命中 | 最终纯净分上限 50 |
| Proxy 或 VPN 单项命中 | 最终纯净分上限 65 |
| Risk ≥ 80 | 额外扣 20 |
| Risk 60～79 | 额外扣 12 |
| Risk 40～59 | 额外扣 5 |

“上限”比简单扣分更适合严重风险。例如原始纯净分为 96，但 Tor 命中后必须限制为 10，而不是只扣 20 后得到 76。

### 7.4 多源一致性修正

多个来源同时命中时提高可信度并加强限制：

```js
const proxyVpnHits = [
  ipapi.isProxy || ipapi.isVpn,
  proxycheck.isProxy || proxycheck.isVpn
].filter(Boolean).length;
```

规则：

```text
0 个来源命中：不处理
1 个来源命中：使用单来源上限
2 个来源命中：纯净分上限 50
```

Tor 或明确 Abuser 不需要多源确认，单来源即可触发硬性上限，但 UI 应显示具体来源。

### 7.5 最终计算顺序

必须固定顺序，避免结果不稳定：

```text
1. 读取 IPPure 基础分
2. 应用 ipapi Abuser 扣分
3. 应用 proxycheck 数值扣分
4. 应用 Proxy/VPN 多源上限
5. 应用 Abuser 上限
6. 应用 Tor 上限
7. clamp 到 0～100
8. 根据最终分确定等级
9. 根据风险信号确定风险标签
```

伪代码：

```js
function calculatePurityRating(sources) {
  if (sources.ippure.ok) {
    score = 100 - sources.ippure.risk;
  } else {
    score = estimateFallbackScore(sources);
  }

  score -= abuserPenalty(sources.ipapi);
  score -= proxyRiskPenalty(sources.proxycheck);

  if (hasMultiSourceProxyVpn(sources)) score = Math.min(score, 50);
  else if (hasSingleSourceProxyVpn(sources)) score = Math.min(score, 65);

  if (hasAbuser(sources)) score = Math.min(score, 25);
  if (hasTor(sources)) score = Math.min(score, 10);

  score = clamp(Math.round(score), 0, 100);

  return buildRatingResult(score, sources);
}
```

## 8. IPPure 失败时的降级方案

### 8.1 禁止默认满分

错误方式：

```js
const fraudScore = data?.fraudScore || 0;
const pureScore = 100 - fraudScore;
```

如果请求失败，以上代码会得到 100 分。

正确方式：

```js
const fraudScore = numberOrNull(data?.fraudScore);
const ippureOk = fraudScore !== null;
```

### 8.2 ipapi 与 proxycheck 都成功

可以生成估算纯净分：

```js
estimatedRisk =
  ipapiAbuserPercent * 0.4 +
  proxycheckRisk * 0.6;

score = 100 - estimatedRisk;
```

UI 必须显示：

```text
估算分
2源
```

而不是显示成 IPPure 实测分。

### 8.3 只有一个补充来源成功

根据单一来源估算，但限制最高分：

```text
最高只能显示 85 分
可信度：低
1源
```

### 8.4 所有来源都失败

不生成数值：

```js
{
  available: false,
  score: null,
  gradeLabel: "待核验",
  riskLabel: "数据不足",
  sourceCount: 0
}
```

UI 显示：

```text
纯净度待核验
暂无有效风险源
```

## 9. 纯净等级与风险标签

### 9.1 纯净等级

| 纯净分 | 等级 | 颜色 |
|---:|---|---|
| 90～100 | S级 | 绿色 |
| 80～89 | A级 | 绿色 |
| 65～79 | B级 | 蓝色或黄色 |
| 50～64 | C级 | 橙色 |
| 0～49 | D级 | 红色 |
| 无有效数据 | 待核验 | 灰色 |

等级函数：

```js
function purityGrade(score) {
  if (score === null) return "待核验";
  if (score >= 90) return "S级";
  if (score >= 80) return "A级";
  if (score >= 65) return "B级";
  if (score >= 50) return "C级";
  return "D级";
}
```

### 9.2 风险标签

风险标签不能只看分数，还应看硬性标记：

| 条件 | 标签 |
|---|---|
| Tor 命中 | 极高风险 |
| Abuser 明确命中 | 高风险 |
| 最终分 < 35 | 高风险 |
| 最终分 35～64 | 中等风险 |
| 最终分 65～79 | 中低风险 |
| 最终分 80～89 | 低风险 |
| 最终分 ≥ 90 | 极低风险 |
| 无有效来源 | 数据不足 |

## 10. 数据可信度

根据有效数据源数量和主来源状态生成可信度：

| 条件 | 可信度 | UI 文案 |
|---|---|---|
| IPPure 成功，且另外两个来源成功 | 高 | `3源` |
| IPPure 成功，另一个来源成功 | 中高 | `2源` |
| IPPure 单独成功 | 中 | `IPPure` |
| IPPure 失败，两个补充来源成功 | 中 | `估算·2源` |
| IPPure 失败，仅一个来源成功 | 低 | `估算·1源` |
| 全部失败 | 无 | `待核验` |

可信度只表示信息可靠程度，不直接增加或减少纯净分。

## 11. IP 属性判断

IP 属性与风险完全分开。

建议优先级：

```text
1. IPPure isResidential === true → 住宅IP
2. 多来源 mobile 命中          → 移动网络
3. 多来源 datacenter/hosting    → 商业机房
4. IPPure isResidential === false → 非住宅IP
5. 无法确认                    → 未知属性
```

禁止以下推断：

```text
Proxy → 住宅IP
VPN   → 住宅IP
低风险 → 住宅IP
机房IP → 高风险
```

建议类型对象：

```js
{
  type: "residential",
  label: "住宅IP",
  tone: "green",
  source: "IPPure"
}
```

## 12. ISP 信息处理

ISP 不作为纯净等级的直接决定因素，但需要正确展示。

### 12.1 ISP 数据优先级

```text
1. ipapi.co org
2. 出口多源合并后的 ISP
3. proxycheck provider/organisation
4. ipapi.is company.name / asn.name
5. 未知ISP
```

### 12.2 ISP 名称规范化

保留运营商别名：

```text
CMCC / China Mobile → 中国移动
China Telecom / Chinanet → 中国电信
China Unicom → 中国联通
CBN / Broadcast → 中国广电
Amazon / AWS → AWS
Microsoft / Azure → Microsoft Azure
```

短关键词不能直接进行普通子字符串匹配。

建议：

```js
function matchProvider(input, aliases) {
  // 先精确匹配，再按关键词长度从长到短匹配。
  // 小于 4 个字符的别名只允许词边界匹配。
}
```

### 12.3 ISP 在 UI 中的作用

用于显示：

```text
DEBEE · 香港 · 70ms
AWS · 东京 · 120ms
HKT · 香港 · 35ms
```

不显示主观的“ISP 低风险”结论。

## 18. 请求策略与性能

### 18.1 超时

建议：

```text
IPPure：3500ms
ipapi.is：4000ms
proxycheck.io：4000ms
```

### 18.2 并发

三个风险请求必须并发，不能依次等待。

### 18.3 缓存

可以新增模块级缓存：

```js
let purityCache = null;
```

缓存键：

```text
出口 IP
```

建议缓存时间：

```text
10～20 分钟
```

出口 IP 未变化时复用风险结果；延迟与服务检测仍按原逻辑刷新。

### 18.4 请求一致性

IPPure 请求必须确保与当前代理出口使用相同策略，否则可能检测到本地直连 IP。

当环境变量 `POLICY` 存在时，IPPure、ipapi.is 和 proxycheck.io 应统一使用该策略。

## 20. 验收标准

以下条件全部满足才算完成：

1. IPPure 成功时，基础纯净分严格等于 `100 - fraudScore`。
2. IPPure 失败时不会产生默认 100 分。
3. `abuser_score` 可以正确解析小数、百分数和等级文本。
4. Tor 命中后最终分不高于 10。
5. 明确 Abuser 命中后最终分不高于 25。
6. Proxy/VPN 不再被当作住宅 IP 证据。
7. 延迟、IPv6、QUIC、NAT、DNS 和服务解锁不影响纯净等级。
8. 底部 UI 明确展示纯净分、等级、风险、来源数、IP 属性和 Abuser 状态。
9. API 全部失败时显示“待核验”。
10. UI 在 Egern Medium Widget 中没有文本溢出和异常行距。
11. JavaScript 通过语法检查。
12. 原有流媒体、AI、延迟、出口检测功能不发生回归。

## 23. 最终决定

本方案最终采用：

```text
IPPure 决定基础纯净分
ipapi.is 修正 Abuser 风险
proxycheck.io 修正 Proxy/VPN/Tor 风险
严重风险使用分数上限，不使用简单平均
ISP 与 IP 属性用于解释，不直接代替纯净度
网络和服务作为辅助摘要，不参与纯净等级
```

底部板块的核心含义从原来的“混合节点综合分”调整为：

```text
IP 纯净度评级卡
```

这样可以确保分数来源明确、等级可解释、数据失败不误判，同时保留 NetworkRadar Pro 原有的网络与服务检测优势。
